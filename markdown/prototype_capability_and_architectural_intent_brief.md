# Prototype Capability & Architectural Intent Brief

**Subject:** `fraud-realtime-ml-prototype`
**Date:** 2026-04-27
**Scope:** Code-grounded analysis — no speculation, no roadmap commentary

---

## 1. Purpose & Intent

This prototype demonstrates a **self-contained, end-to-end real-time fraud scoring platform** that spans from synthetic data generation through feature engineering, model training, and sub-100ms online inference.

The system is designed to prove that a single codebase can:

- Serve fraud scores in real-time via a REST API backed by an XGBoost model and 37+ features drawn from three retrieval paths (request payload, batch feature store, real-time sliding windows).
- Maintain **training-serving consistency** by logging the exact online features observed at inference time and replaying them during model training.
- Support **config-driven multi-model experimentation** (XGBoost, LightGBM, RandomForest) with calibration, preprocessing, and temporal evaluation — without code changes.
- Orchestrate the full offline pipeline (Postgres → DuckDB → dbt → Parquet → Feast → Redis) through a single Makefile.

The prototype operates entirely on synthetic data generated by its own simulators. There is no integration with any production data source, external API, or cloud service.

## 2. Targeted Legacy Pain Points

The following pain points are **directly addressed by implemented code**, not by aspirational design documents.

| Pain Point | How the Prototype Addresses It | Evidence |
|---|---|---|
| **Feast SDK serving latency (~15–20ms)** | `feast_direct.py` bypasses the Feast Python SDK entirely. It reads Redis binary keys directly using `mmh3` field hashes and inline protobuf decoding, achieving ~2ms per fetch. | `app/feast_direct.py`: `_build_entity_key()`, `_decode_proto_value()`, `_compute_field_hashes()` |
| **Training-serving feature skew** | `feature_logger.py` captures every inference's actual online feature values into `online_feature_log`. `build_training_dataset.py` joins this table by `transaction_id` so the training set contains the real features the model saw, not SQL approximations. | `app/feature_logger.py`, `training/build_training_dataset.py`, `sql/migrations/03_add_online_feature_log.sql` |
| **Cold-start training (no inference history)** | Before `online_feature_log` accumulates data, dbt models tagged `cold_start` (`int_user_txn_online_stats`, `int_device_txn_online_stats`, `int_user_login_online_stats`) compute SQL-based approximations of the sliding-window features using timestamp self-joins. `build_training_dataset.py` falls back to these when the log table is empty. | `dbt_project/models/intermediate/int_user_txn_online_stats.sql`, `training/build_training_dataset.py` |
| **Composite-key cache thrashing** | Per-entity TTL caches (separate `TTLCache` instances for user, device, merchant) replace a single composite-key cache. A returning user hitting a new merchant still gets cache hits for user and device features. | `app/feature_fetcher.py`: three `TTLCache` instances with 60s TTL |
| **Single-command Redis round trips for online features** | The `retriever.py` module batches all 11 `ZRANGEBYSCORE` commands into a single Redis pipeline — one network round trip regardless of feature count. | `app/online_features/retriever.py`: `async with self._redis.pipeline(transaction=False)` |
| **Uncalibrated model outputs** | The training pipeline supports sigmoid, isotonic, and beta calibration with a carved-out calibration set, ensuring scores are well-calibrated probabilities for downstream risk decisioning. | `training/train_model.py`: `_calibrate_model()`, experiment YAML `calibration:` section |
| **Manual model promotion** | `promote_model.py` copies MLflow artifacts to the serving directory, updates `.env` with the new model path, tags the promoted run as `champion`, and clears previous champion tags — all in one command. | `scripts/promote_model.py` |
| **Blocking inference logging** | Both `score_logger.py` and `feature_logger.py` use async queue-based drain loops (50ms batch interval, 100-row max) so logging never blocks the scoring hot path. | `app/score_logger.py`, `app/feature_logger.py` |

## 3. Architectural Patterns & Design Principles

### 3.1 Dual-Path Feature Architecture

The prototype implements a clear separation between **batch features** (daily/weekly/monthly aggregates computed by dbt and served via Feast/Redis) and **online features** (sub-minute sliding windows maintained in Redis sorted sets). Both paths are fetched in parallel via `asyncio.gather` in the scoring endpoint.

```
Batch path:  Postgres → DuckDB → dbt → Parquet → Feast materialize → Redis (hash keys)
Online path: stream_transactions.py → updater.py → Redis (sorted sets via ZADD)
```

### 3.2 Feast as a Feature Contract Layer, Not a Serving Layer

Feast is used for:
- Feature definitions and versioning (`*_batch_fv_v1`, `fraud_scoring_v1`)
- Offline point-in-time joins during training dataset construction
- Materialization from Parquet to Redis

Feast is **not** used for online serving. The SDK is bypassed at read time by `feast_direct.py`, which reimplements the Redis key format (entity key serialization v2, `mmh3` field hashing, protobuf value decoding).

### 3.3 Feature Versioning

Feature views follow `<entity>_batch_fv_<version>` naming. The feature service `fraud_scoring_v1` pins to specific view versions. A `FEATURE_SERVICE_VERSION` constant is logged with every prediction in `model_score_log`, enabling traceability between scores and feature definitions.

### 3.4 Config-Driven Experimentation

Model training requires zero code changes between experiments. YAML configs in `training/experiments/` control:
- Model type and hyperparameters
- Split strategy (temporal OOT or random stratified)
- Preprocessing (passthrough, standard/minmax/robust scaler, ordinal/one-hot encoding)
- Calibration method and fraction
- Threshold strategy (`recall_target` with minimum recall, or `fixed`)

### 3.5 Non-Blocking Async Logging

Inference logging is decoupled from the scoring path. Both loggers (`score_logger.py`, `feature_logger.py`) enqueue records to an `asyncio.Queue` and drain in background tasks with batched `INSERT` statements via `asyncpg`. The drain loop runs at 50ms intervals with a 100-row batch cap.

### 3.6 Incremental Processing

All dbt intermediate models use `materialized='incremental'` with `unique_key='transaction_id'`. On incremental runs, only transactions with `event_timestamp > max(existing)` are processed, enabling efficient re-runs over growing datasets.

### 3.7 Infrastructure as Configuration

Docker Compose defines the full stack (Postgres 15, Redis 7, FastAPI) with resource limits, health checks, and volume mounts. The Makefile (30+ targets) provides a complete task runner covering setup, data generation, feature engineering, training, serving, and load testing.

## 4. Implemented Capabilities (with Evidence)

### 4.2 Feature Retrieval (Three Sources)

| Source | Features | Mechanism | File |
|---|---|---|---|
| **Request payload** | `txn_amount`, `is_international`, `local_hour` | Extracted from `ScoreRequest` | `app/scoring.py` |

Missing features default to `0` during vector assembly (`app/scoring.py`: `build_feature_vector()`).

### 4.3 Online Feature Management

| Capability | Evidence |
|---|---|

| Member encoding: `{event_id}:{payload}` with float scores as Unix timestamps | `redis_keys.py`: `encode_member()`, `decode_member()` |
| ZADD + ZREMRANGEBYSCORE trim + TTL refresh on every event | `app/online_features/updater.py` |

### 4.4 Offline Feature Engineering (dbt)

| Layer | Models | Responsibility |
|---|---|---|
| Staging (6 models) | `stg_transactions`, `stg_users`, `stg_devices`, `stg_merchants`, `stg_login_events`, `stg_fraud_labels` | Type casting, cleaning, composite indexes |
| Intermediate (7 models) | `int_user_txn_stats`, `int_user_login_stats`, `int_device_stats`, `int_merchant_stats` + 3 `cold_start` models | 1d/7d/30d rolling windows; cold-start SQL approximations of 5m/10m/1h windows |
| Features (3 models) | `fct_user_features`, `fct_device_features`, `fct_merchant_features` | Final feature tables joining intermediate aggregates |

All intermediate models use incremental materialization. The `cold_start` models use self-joins with `event_timestamp < anchor_timestamp` for point-in-time correctness.

### 4.5 Training Pipeline

| Stage | File | Capabilities |

## 6. Known Limitations & Risks

### 6.1 Operational Limitations

| Limitation | Impact | Evidence |
|---|---|---|
| **Single-model serving** | No hot-reload, no A/B testing, no fallback model. Model is loaded once at startup. A deployment requires a full API restart. | `app/model_loader.py`: global `_model`, `_preprocessor`, `_calibrated_model` |
| **SQLite-backed MLflow** | Not suitable for concurrent multi-user experiment tracking. File locking can cause issues under parallel training runs. | `mlflow.db` (SQLite), `.env`: `MLFLOW_TRACKING_URI=sqlite:///mlflow.db` |
| **Feast SDK bypass is tightly coupled to Feast internals** | `feast_direct.py` depends on Feast's entity key serialization format (v2), `mmh3` field hashing, and protobuf value encoding. A Feast version upgrade could silently break binary compatibility. | `app/feast_direct.py`: hardcoded struct format `<IBQ`, mmh3 hash seeds |
| **No graceful shutdown for loggers** | The async drain loops in `score_logger.py` and `feature_logger.py` run as background tasks. If the API process is killed abruptly, queued records are lost. | `app/score_logger.py`, `app/feature_logger.py`: `asyncio.Queue` with no persistence |
| **Default-to-zero for missing features** | `build_feature_vector()` defaults missing features to `0`. For features where `0` is a valid and meaningful value (e.g., `txn_count_5m = 0` means "no recent transactions"), this is correct. For others, it may mask data retrieval failures. | `app/scoring.py`: `features.get(col, 0)` |
| **Resource limits assume local development** | Docker Compose allocates 2.5 CPU / 2GB to the API. Production workloads at higher concurrency would require different sizing. | `docker-compose.yml`: `deploy.resources` |

### 6.2 Test Coverage Gaps

| Untested Area | Risk |
|---|---|
| Training pipeline (`build_training_dataset.py`, `train_model.py`, `evaluate_model.py`) | Regression in dataset construction or training logic would go undetected |
| Async drain loops (`score_logger.py`, `feature_logger.py`) | Batch insert failures, queue overflow, or connection pool exhaustion untested |
| Feature caching logic (`feature_fetcher.py` TTLCache paths) | Cache miss/partial-hit/expiry edge cases untested |
| Model loading priority (`model_loader.py` calibrated vs base fallback) | Incorrect artifact resolution could silently serve the wrong model |
| Scripts (`export_pg_to_duckdb.py`, `materialize_features.py`, `promote_model.py`) | Pipeline breaks in offline data flow would require manual debugging |

### 6.3 Security Considerations

| Risk | Detail |
|---|---|
| **No API authentication** | All endpoints are publicly accessible. Suitable for local development only. | 
| **Database credentials in `.env`** | Default credentials (`fraud_user`/`fraud_pass`) are committed in `.env.example`. The `.env` file is not gitignored in the actual repo (present in tree). |
| **No input sanitization beyond Pydantic** | Pydantic validates types and required fields. No additional validation on value ranges (e.g., negative `txn_amount`) beyond what the schema enforces. |

### 6.4 Architectural Risks

| Risk | Detail |
|---|---|
| **Feast version lock-in** | The `feast_direct.py` bypass reimplements Feast's internal binary format. This creates an implicit dependency on Feast 0.40.0's serialization behavior that is not covered by Feast's public API contract. |
| **DuckDB as offline store** | DuckDB is single-process. Concurrent dbt runs or training dataset builds against the same `.duckdb` file will fail. Suitable for single-developer workflows only. |
| **No feature backfill strategy** | If Redis is flushed or a feature view schema changes, there is no automated mechanism to backfill online features. `materialize_features.py` handles batch features but not sorted-set online features. |
| **Synthetic data only** | All simulators generate synthetic data. The fraud patterns are plausible but unvalidated against real-world distributions. Model performance on synthetic data does not predict production performance. |
